# Patch Candidate Schema

This document defines the JSONL records used by the patch-verification study.
The schema is intentionally explicit so that later scripts can validate it
without model calls.

## Candidate Record

Each candidate patch is one JSON object.

Required fields:

```json
{
  "patch_id": "bugsinpy_httpie_2__reference_fix",
  "model_candidate_id": "candidate_0001",
  "task_id": "bugsinpy_httpie_2",
  "project": "httpie",
  "source": "bugsinpy",
  "candidate_type": "correct_reference",
  "expected_outcome": "correct",
  "base_version": "buggy",
  "patch_text": "diff --git ...",
  "touched_files": ["httpie/client.py"],
  "context_mode": "patch_hunks",
  "visible_context": "...",
  "issue_summary": "max redirects should be assigned before request execution",
  "visible_tests": ["tests/test_redirects.py::TestRedirects::test_max_redirects"],
  "hidden_oracles": ["scripts/oracles/httpie_2_max_redirects.py"],
  "oracle_command": "python scripts/oracles/httpie_2_max_redirects.py",
  "oracle_workdir": "data/real_bugs/bugsinpy_workspace/httpie_2/patched/httpie",
  "evidence_sources": ["regression_oracle", "patch_diff"],
  "label_confidence": "high",
  "construction_notes": "Reference fix from BugsInPy patch.",
  "patch_materialization": "buggy_fixed_unified_diff"
}
```

`patch_id` is evaluator-facing and may encode construction provenance. It must
not be shown to a model. `model_candidate_id` is the anonymous identifier used
inside model-visible evidence packets.

## Candidate Types

Use these values for `candidate_type`:

- `correct_reference`: known reference fix.
- `buggy_noop`: original buggy code or a no-op patch.
- `partial_fix`: addresses part of the behavior but leaves another checked
  behavior broken.
- `overfitted_fix`: passes a visible check but fails a stronger hidden oracle.
- `irrelevant_patch`: changes unrelated code.
- `model_generated_patch`: generated by a coding model or agent.
- `human_constructed_negative`: manually constructed negative candidate with a
  documented rationale.

## Expected Outcomes

Use these values for `expected_outcome`:

- `correct`: should be accepted.
- `incorrect`: should be rejected.
- `partial`: should be rejected or escalated.
- `overfitted`: should be rejected or escalated.
- `irrelevant_or_noop`: should be rejected.
- `environment_invalid`: cannot be used for evaluation.

## Patch Materialization

Use `patch_materialization` to record how `patch_text` was created. This is an
evaluator-facing provenance field and must not be shown to a model.

Current values:

- `buggy_fixed_unified_diff`: diff generated by comparing retained buggy and
  fixed checkouts.
- `empty_diff_against_buggy_checkout`: empty diff control against the buggy
  checkout.
- `local_comment_only_unified_diff`: applicable comment-only source diff used
  as an unrelated negative control.
- `first_hunk_of_reference_unified_diff`: partial candidate generated from the
  first hunk of a multi-hunk reference diff.
- `reference_diff_with_one_change_omitted`: partial candidate generated by
  applying a retained reference diff except one change block.
- `reference_replace_with_one_line_reverted`: partial candidate generated by
  reverting one line inside a retained multi-line replace block.

## Verification Record

Verifier outputs should be stored separately from candidate records.

```json
{
  "patch_id": "bugsinpy_httpie_2__reference_fix",
  "verifier_id": "llm_only_gpt",
  "condition": "llm_only",
  "decision": "accept",
  "confidence": 0.82,
  "claims": [
    {
      "claim_id": "claim_001",
      "claim": "The patch assigns max_redirects to the session before request execution.",
      "evidence": ["line 42", "visible regression test name"],
      "evidence_status": "claimed_by_model"
    }
  ],
  "raw_response_path": "outputs/patch_verification_pilot_001/reviews.jsonl",
  "cost_usd": 0.0,
  "invalid_reason": null
}
```

Allowed `decision` values:

- `accept`;
- `reject`;
- `escalate`;
- `invalid_output`.

## Model-Visible Evidence Packet

Evidence packets are the only records passed to reviewers. They intentionally
omit evaluator labels, construction notes, hidden oracle paths, and provenance
encoded in `patch_id`.

```json
{
  "candidate_id": "candidate_0001",
  "task_id": "bugsinpy_httpie_2",
  "project": "httpie",
  "task_summary": "max redirects should be assigned before request execution",
  "visible_context": "...",
  "patch_text": "diff --git ...",
  "touched_files": ["httpie/client.py"],
  "visible_tests": ["tests/test_redirects.py::TestRedirects::test_max_redirects"],
  "available_evidence_sources": ["regression_oracle", "patch_diff"],
  "has_hidden_oracle": true,
  "evidence_packet_version": "patch_evidence_packet_v1",
  "label_leakage_guard": "ground-truth labels and oracle results are intentionally omitted"
}
```

## Label Integrity Rules

- `expected_outcome` must never be shown to a model.
- `patch_materialization` must never be shown to a model.
- Evaluator-facing `patch_id` must never be shown to a model when it encodes
  candidate construction type.
- `hidden_oracles` may be used by the evaluator, but the model should only see
  visible context configured for the condition.
- Any manually constructed patch must include `construction_notes`.
- Any model-generated patch must include model id, prompt version, generation
  date, and decoding settings in a companion metadata record.
- Any candidate without executable or tool-backed evidence is not valid for the
  main experiment.

## Task-Level Generation Accounting

Each expanded task should also have a task metadata record. This record
separates task validity from generator success.

Required fields for expanded studies:

```json
{
  "task_id": "bugsinpy_httpie_5",
  "project": "httpie",
  "bug_id": "5",
  "environment_stable": true,
  "reference_patch_valid": true,
  "buggy_failure_reproducible": true,
  "pass_to_pass_stable": true,
  "validation_runtime_seconds": 12.4,
  "generator_attempts": 4,
  "num_ai_patches_generated": 3,
  "num_ai_patches_applicable": 1,
  "num_ai_patches_correct": 0,
  "num_ai_patches_incorrect": 1,
  "generation_status": "unsolved",
  "task_role": "hard_generation_case",
  "main_experiment_included": true,
  "exclusion_reason": null
}
```

Allowed `generation_status` values:

- `solved`;
- `partially_solved`;
- `unsolved`;
- `non_applicable_only`;
- `not_attempted`;
- `not_applicable`.

Allowed `task_role` values:

- `main_balanced_task`;
- `hard_generation_case`;
- `stress_case`;
- `appendix_case`;
- `excluded_unstable_validation`.

Rules:

- A task may remain in the main experiment when the generator fails, as long as
  validation is stable and reference labels are reliable.
- A task should be excluded only when reproducible validation criteria fail.
- Exclusion because a model failed to solve a task is not allowed.
- Non-applicable generated patches may be retained as a failure type, but their
  share should be capped in the main experiment.
